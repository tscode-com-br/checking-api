Fluxo com 2 leitores RC522:
1. O leitor RC522 #1 envia `action=checkin` para /api/scan.
2. O leitor RC522 #2 envia `action=checkout` para /api/scan.
3. A API executa a acao recebida e atualiza `users.checkin` apenas quando o envio ao Forms for bem-sucedido.
4. RFID desconhecido entra em pendencia na aba Cadastro.
5. Quando a API responder que o RFID nao esta cadastrado, o LED interno acende laranja por 4 segundos.
6. Assim que o laranja apagar, a ESP32 volta imediatamente ao estado online, pisca o LED branco e aceita nova leitura de RFID.
7. Se o usuario ja estiver em check-in, a API apenas atualiza o campo `local` e a ESP32 pisca o LED verde 3 vezes em 1 segundo antes de voltar ao branco de prontidao.
8. Para usuario cadastrado fora de check-in, e a propria API que preenche o Forms e busca cada elemento obrigatorio em etapas de ate 10 segundos.
9. No checkout, se o usuario nao tiver check-in ativo, a ESP32 acende o LED vermelho por 2 segundos e volta ao estado de prontidao.
10. Se qualquer etapa obrigatoria falhar, a ESP32 pisca o LED vermelho 5 vezes em 1 segundo e volta ao estado de prontidao.
11. Quando o envio ao Forms for concluido com sucesso, a ESP32 acende o LED verde por 1 segundo e volta ao estado online.
12. Quando um usuario realiza check-in com sucesso, ele deixa de aparecer na aba Check-Out do admin.

Auditoria da API na aba Eventos:
1. A API registra heartbeats, scans recebidos, pendencias, bloqueios, tentativas e resultados do envio ao Forms.
2. Cada registro pode incluir origem, device, local, RFID, projeto, rota, codigo HTTP, tentativas, mensagem e detalhes.

Fluxo de boot e monitoramento da nuvem:
1. Ao energizar, o LED interno da ESP32 acende amarelo imediatamente.
2. A ESP32 tenta conectar ao Wi-Fi e enviar heartbeat para a API.
3. Quando recebe resposta positiva da API, o LED branco passa a piscar 1 vez por segundo.
4. A ESP32 repete o heartbeat a cada 3 minutos.
5. Em falha de conectividade ou resposta negativa, o LED vermelho fica aceso continuamente.
6. Enquanto o LED vermelho estiver aceso por indisponibilidade, a leitura de cartoes fica bloqueada.
7. Se a indisponibilidade permanecer por 30 segundos, a ESP32 reinicia para repetir a tentativa de conexao ao Wi-Fi e ao sistema.

Ligacao SPI dos dois RC522:
1. ESP32 GPIO 12 -> SCK dos dois RC522.
2. ESP32 GPIO 11 -> MOSI dos dois RC522.
3. ESP32 GPIO 13 -> MISO dos dois RC522.
4. ESP32 GPIO 9 -> RST dos dois RC522.
5. ESP32 GPIO 10 -> SDA/SS do RC522 #1.
6. ESP32 GPIO 15 -> SDA/SS do RC522 #2.
7. 3V3 -> VCC e GND comum para ambos.

Projeto Android atual em Flutter (checking_android_new):
1. O projeto mobile atual fica em .\checking_android_new.
2. O emulador configurado neste computador e o Pixel_7_API_35.

Rodar o app Android com um clique no VS Code:
1. Abrir Terminal > Run Task.
2. Selecionar a task Run Checking Android App.
3. A task abre o emulador Pixel_7_API_35 se ele ainda nao estiver aberto.
4. Depois a task entra em .\checking_android_new e executa flutter run no emulador ativo.
5. Para parar a execucao, usar Ctrl+C no terminal da task ou Terminal > Kill Task.

Rodar o app Android manualmente por terminal:
1. Definir variaveis do Android SDK:
	$env:ANDROID_HOME="$env:LOCALAPPDATA\Android\Sdk"
	$env:ANDROID_SDK_ROOT="$env:LOCALAPPDATA\Android\Sdk"
	$env:Path="$env:ANDROID_HOME\platform-tools;$env:ANDROID_HOME\emulator;$env:Path"
2. Entrar na pasta do projeto Android:
	cd .\checking_android_new
3. Rodar o app no emulador:
	flutter run -d emulator-5554
